## Overview

The `LDAP::Server::Connection` class manages individual client connections. It handles the asynchronous nature of LDAP, where multiple requests and responses can be in flight simultaneously. Each request is processed in its own thread, with a mutex protecting the I/O stream.

## Instance Methods

### `handle_requests`

Main request handling loop. Reads LDAP messages from the socket, decodes them, and dispatches to appropriate handlers. This method blocks until the connection is closed.

```ruby
LDAP::Server::Connection.new(socket, options).handle_requests
```

Handles these LDAP operations:
- **BindRequest (tag 0)**: Authentication
- **UnbindRequest (tag 2)**: Closes connection
- **SearchRequest (tag 3)**: Directory searches
- **ModifyRequest (tag 6)**: Entry modifications
- **AddRequest (tag 8)**: Add new entries
- **DelRequest (tag 10)**: Delete entries
- **ModifyDNRequest (tag 12)**: Rename/move entries
- **CompareRequest (tag 14)**: Attribute comparison
- **AbandonRequest (tag 16)**: Cancel pending operations

### `startssl`

Upgrades the connection to SSL/TLS. Yields to a block before performing the upgrade if provided.

```ruby
connection.startssl do
  # Optional: send StartTLS extended response
end
```

### `ber_read(io)`

<ResponseField name="Returns" type="String">

<ResponseField name="Throws">

</ResponseField>

### `write(data)`

Writes data to the socket in a thread-safe manner.

<ParamField path="data" type="String">


### `writelock`

Yields the IO object with exclusive write access, then flushes.

```ruby
connection.writelock do |io|
  io.write(response1)
  io.write(response2)
end
```

### `abandon(messageID)`

Abandons a pending operation by raising `LDAP::Abandon` in its thread.

<ParamField path="messageID" type="Integer">
  Message ID of the operation to abandon
</ParamField>

```ruby

```

### `abandon_all`

Abandons all pending operations. Called when the client unbinds or disconnects.

```ruby

```

## Thread Safety

The Connection class uses a mutex to ensure thread-safe access to the I/O stream. Multiple operations can run concurrently in separate threads, but responses are serialized through the mutex to prevent interleaving.

## Example

The Connection class is typically instantiated by the Server:

```ruby
# In server.rb
LDAP::Server.tcpserver(options) do
  LDAP::Server::Connection.new(self, opt).handle_requests
end
```
